# 백엔드 테스트 규칙

## 적용 범위
- `apps/server` 하위의 Go 백엔드 변경 전체에 적용한다.

## 필수 개발 방식
- 백엔드 변경은 항상 TDD 순서로 진행한다.
- 기본 순서: `Fail -> Pass -> Refactor`
- 테스트 없이 비즈니스 로직을 먼저 확장하지 않는다.

## 커버리지 기준
- 백엔드 전체 커버리지는 `100%`를 유지한다.
- 기준 명령은 `cd apps/server && go test -p 1 ./... -coverprofile=/tmp/lifebase-cover.out` 이다.
- 패키지 단위가 아니라 전체 직렬 실행 기준으로 검증한다.

## 테스트 구조
- Go 유닛/통합 테스트는 대상 코드와 같은 패키지 디렉토리에 `*_test.go`로 둔다.
- 별도 루트 `test/` 미러 디렉토리는 기본 구조로 채택하지 않는다.
- 공통 테스트 도우미는 `apps/server/internal/testutil`에 둔다.

## 설계 원칙
- 테스트 편의를 위한 변경은 허용하되, 공개 계약을 훼손하지 않는다.
- 외부 의존성(DB, 시간, 락, HTTP, 워커)은 작은 hook/helper로 분리해 테스트 가능성을 높인다.
- 헥사고날 경계를 유지한다. Controller에 비즈니스 로직을 넣거나 UseCase를 우회하는 테스트 전용 구조를 만들지 않는다.

## 검증 원칙
- 예외 분기, DB 에러 분기, `rows.Err()`, `Scan` 실패, 외부 API 실패 분기까지 포함해 검증한다.
- 실행이 불안정한 패키지는 원인을 제거하거나 구조를 개선한다. flaky 테스트를 허용하지 않는다.
- 커버리지 100% 미만이면 커밋 완료로 보지 않는다.
