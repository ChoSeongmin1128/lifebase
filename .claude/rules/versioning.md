# 버전 관리 규칙

## 단계별 체계

### 프로토타입 단계 (현재)
- `v1`, `v2`, `v3` 단순 정수 버전 사용
- 프로토타입 HTML 파일의 `<title>`과 헤더에 표기
- 의미 있는 UI/UX 변경 묶음 단위로 올림

### 구현 단계
- Semantic Versioning `MAJOR.MINOR.PATCH` 사용
- 최초 구현 시작은 `0.1.0`부터

## 버전 올림 기준

### MAJOR (X.0.0)
- 하위 호환이 깨지는 변경
- 예: API 응답 구조 변경, DB 스키마 비호환 마이그레이션, 인증 방식 변경

### MINOR (0.X.0)
- 하위 호환을 유지하면서 기능 추가/변경
- 예: 새 탭/뷰 추가, 동기화 방식 추가, 새 API 엔드포인트

### PATCH (0.0.X)
- 기존 기능의 버그 수정, 성능 개선, 문구/스타일 수정
- 예: 레이아웃 깨짐 수정, 동기화 오류 수정, 오타 수정

## 0.x 구간 규칙
- `0.x.y` 동안은 MINOR 변경에도 하위 호환 깨짐 허용
- `1.0.0`은 첫 공개 릴리스 시점에 선언

## 버전 관리 위치
- 단일 소스: 루트 `package.json`의 `version` 필드
- 태그: 릴리스 시 `git tag v0.1.0` 형식으로 생성
- 프로토타입 파일의 단순 버전(v1, v2 …)은 구현 단계 진입 시 제거

## 릴리스 체크순서
1. 변경 영향(MAJOR/MINOR/PATCH) 판단
2. 루트 `package.json` 버전 갱신
3. 문서 내 버전 표기(`docs/700-마일스톤.md` 등) 동기화
4. 태그 생성 전 테스트/빌드 확인
5. 태그 생성 및 릴리스 노트 기록

## 드리프트 방지
- 코드 버전과 문서 버전이 다르면 릴리스를 중단하고 먼저 동기화한다.
- 버전 표기를 담는 `README.md`, `docs/700-마일스톤.md`, `plan.md`는 같은 작업에서 함께 점검한다.
- API 수, 마이그레이션 수, 플랫폼 상태처럼 버전 판단에 영향을 주는 요약 문서도 코드 상태와 함께 맞춘다.
