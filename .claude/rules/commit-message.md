# 커밋 메시지 규칙

## 형식
```text
<type>: <한글 요약>

- 수정 내용 1
- 수정 내용 2
```

## 제목 규칙
- `<type>`은 영문 소문자
- `:` 뒤 한 칸 + 한글 요약
- 한 줄 제목, 마침표 생략

## 본문 규칙
- 본문 생략 금지, 한글로 작성
- `- ` 불릿으로 작성
- 각 항목은 "무엇을 왜 바꿨는지"를 포함
- 명사형 종결 (예: `- OAuth 토큰 갱신 로직 추가`, `- 불필요한 의존성 제거`)

## type 목록
- `feat`, `fix`, `refactor`, `style`, `docs`, `test`, `chore`, `perf`, `ci`, `build`, `revert`

## 금지
- `update`, `수정`, `작업` 같은 모호한 제목
- type 없는 제목
- `Co-Authored-By` 등 자동 생성 서명 추가 금지

## 멀티워크트리 통합 규칙
- 플랫폼별 작업 커밋은 브랜치에서 명확히 분리한다.
- 통합 브랜치에서는 web-first 병합 순서를 유지한다.
- 통합 커밋 본문에는 병합 순서(Web -> Desktop/Mobile)와 검증 범위를 기록한다.

## squash merge 기록 규칙
- `dev` 병합은 squash merge를 기본으로 한다.
- PR 설명은 상세 작업 기록의 기준 문서로 관리한다.
- PR 설명에는 배경, 변경 범위, 영향 플랫폼, 테스트/검증 결과, 미실행 항목과 이유, 리스크를 남긴다.
- squash merge 커밋 본문은 최종 의사결정과 검증 범위를 요약해 남긴다.
- 중간 커밋 제목/본문을 squash 커밋에 그대로 나열하지 않는다.
- squash merge 커밋 본문에는 무엇을 왜 바꿨는지와 최종 검증 결과를 함께 기록한다.
