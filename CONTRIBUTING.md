# LifeBase 협업 워크플로우

## 목적
- `dev` 기준 브랜치 운영, git worktree 사용 기준, squash merge 기록 방식을 팀 공통 규칙으로 고정한다.
- 작은 작업과 큰 작업의 기준을 문서화해 작업 시작 전에 판단 차이를 줄인다.

## 기본 원칙
- 모든 작업 브랜치는 `dev` 최신 상태를 기준으로 만든다.
- 병렬 작업 또는 멀티플랫폼 작업은 git worktree 사용을 기본으로 한다.
- 최종 병합 대상 브랜치는 `dev`로 고정한다.
- `dev` 병합은 squash merge를 기본으로 한다.
- Web, Desktop, Mobile에 공통 영향이 있으면 Web을 먼저 구현하고 검증한다.

## 작업 분류 기준

### 작은 작업
- 아래 조건을 모두 만족해야 한다.
- 단일 목적이며 독립 PR 1개로 짧게 끝낼 수 있다.
- 사실상 단일 플랫폼 범위에서 끝난다.
- 변경 범위가 좁고 파일/모듈 충돌 가능성이 낮다.
- DB 스키마, API 계약, 라우팅, 인증/권한, 데이터 접근, 공유, 동기화, 로깅, 파일 처리 정책에 영향이 없다.
- 예: 문구 수정, 단일 컴포넌트 스타일 수정, 로컬 유틸 버그 수정, 테스트 보강, 문서 수정
- 운영 방식: 일반 브랜치 작업 가능, 필요 시 worktree 선택 가능

### 큰 작업
- 아래 조건 중 하나라도 만족하면 큰 작업으로 본다.
- 독립 PR로 볼 수 있는 기능/정책 단위다.
- Web/Desktop/Mobile 중 2개 이상 플랫폼에 영향이 있다.
- DB 스키마, API 계약, 라우팅, 인증/권한, 데이터 접근, 공유, 동기화, 로깅, 파일 처리 정책에 영향이 있다.
- 공통 패키지 또는 여러 모듈을 동시에 수정해 충돌 가능성이 높다.
- 운영 방식: git worktree 사용 기본

### 애매한 작업
- 작은 작업/큰 작업 판정이 불명확하면 추정하지 않는다.
- 영향 범위를 먼저 확인하고 필요하면 작업 시작 전에 기준을 합의한다.

## git worktree 운영
- 브랜치 네이밍 기본 형식은 `task/<scope>-<platform>`이다.
- 외부 작업 ID가 있을 때만 `task/<ticket>-<scope>-<platform>` 형식을 사용한다.
- 예: `task/calendar-sync-web`, `task/workflow-docs-repo`, `task/128-calendar-sync-web`
- Codex는 저장소 상위 디렉터리에서 시작할 수 있다.
- 이 경우 에이전트는 원본 repo 경로를 찾은 뒤 그 repo에서 worktree를 생성한다.
- worktree 생성 후에는 생성된 worktree 경로에서만 구현, 검증, 커밋, PR 작업을 이어간다.
- worktree 디렉터리는 현재 작업 목적이 드러나는 이름으로 만든다.
- worktree는 저장소 루트에서 생성해도 되고, 경로는 보통 저장소 상위 디렉터리의 형제 폴더로 둔다.
- 예: `/Users/seongmin/project/lifebase`에서 `git worktree add ../lifebase-calendar-sync-web -b task/calendar-sync-web dev`
- 작업 브랜치는 생성 직후 `dev` 기준으로 맞추고, 오래 열어두지 않는다.
- 장기 작업은 중간 병합 대신 `git fetch` 후 `dev` 기준 rebase를 우선한다.

## 권장 흐름
1. `dev` 최신화 후 작업 범위를 작은 작업/큰 작업으로 분류한다.
2. 큰 작업이거나 병렬 작업이면 worktree를 만든다.
3. Web 공통 범위를 먼저 구현하고 검증한다.
4. Desktop/Mobile은 Web 결과 기준으로 rebase 후 반영한다.
5. 통합 검증을 마친 뒤 `dev`에 squash merge 한다.

## 에이전트 수행 범위
- 사용자가 작업 완료까지 요청한 경우 에이전트는 아래 범위를 끝까지 수행한다.
- worktree 전략이면 원본 repo에서 worktree 생성 후 생성된 worktree 경로로 작업 컨텍스트 전환
- 작업 브랜치 정리 및 커밋
- 원격 브랜치 푸시
- PR 생성과 본문 작성
- squash merge
- 원격 브랜치 삭제
- 로컬 브랜치와 worktree 정리
- 실패 시 중단 지점, 원인, 필요한 후속 액션을 바로 보고한다.

## PR 작성 규칙
- squash merge를 전제로 하므로 PR이 상세 작업 기록의 기준 문서가 된다.
- PR 설명에는 최소한 아래 내용을 포함한다.
- 배경과 문제 정의
- 변경 범위와 비범위
- 영향 플랫폼과 적용 순서
- 실행한 테스트와 수동 검증 결과
- 미실행 테스트와 그 이유
- 주요 리스크, 롤백 포인트, 후속 작업

## PR 본문 템플릿
```md
## 배경
- 이 변경이 필요한 이유

## 변경 범위
- 이번 PR에 포함한 내용

## 비범위
- 이번 PR에서 의도적으로 제외한 내용

## 영향 플랫폼
- Web:
- Desktop:
- Mobile:

## 검증
- 실행한 테스트:
- 수동 검증:
- 미실행 항목과 이유:

## 리스크와 후속 작업
- 현재 리스크:
- 후속 작업:
```

## squash merge 커밋 규칙
- squash merge 커밋은 최종 의사결정과 검증 범위를 남기는 요약본으로 작성한다.
- 중간 커밋 목록을 본문에 그대로 복붙하지 않는다.
- 커밋 본문에는 최소한 아래 내용을 남긴다.
- 무엇을 바꿨는지
- 왜 바꿨는지
- 병합 순서가 있었으면 그 순서
- 최종 검증 범위와 결과

## squash merge 커밋 예시
```text
feat: 캘린더 화면 통합 흐름 개선

- Web 기준 렌더링 흐름 정리로 공통 상태 전환 일관성 확보
- Desktop 반영 시 Web 변경 기준 리베이스 적용으로 플랫폼 동작 차이 축소
- Mobile 반영 시 네비게이션 분기 조정으로 동일 사용자 흐름 유지
- 병합 순서 Web -> Desktop -> Mobile 적용
- 검증 범위로 Web/Desktop/Mobile 진입 및 핵심 시나리오 점검
```
