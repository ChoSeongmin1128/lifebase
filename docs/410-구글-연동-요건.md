# 구글 연동 요건

## 목적
- Google Calendar/Tasks API 연동의 구현 요건과 API 특성을 정리한다.

## 필수 요건
- OAuth 2.0 기반 사용자 승인
- Calendar 이벤트 CRUD 동기화
- Tasks Todo CRUD 동기화
- 다중 Google 계정 동시 연결 및 계정 단위 장애 격리
- 동기화 실패 재시도(지수 백오프 + ±25% 랜덤 간격)
- 충돌 발생 시 사용자 선택 가능 상태로 전환

## Google Calendar API

### Event 주요 필드
- 기본: summary, description, location, status(confirmed/tentative/cancelled)
- 시간: start/end(date 또는 dateTime + timeZone), 종일 일정은 date 사용
- 색상: colorId(1~11), 미설정 시 캘린더 색상 상속
- 반복: recurrence[](RRULE/EXDATE 등 RFC 5545)
- 알림: reminders.useDefault 또는 overrides[](method + minutes, 최대 5개)
- 참석자: attendees[](email, responseStatus 등)
- 첨부: attachments[](fileUrl, title, mimeType, 최대 25개)
- 가시성: visibility(default/public/private/confidential)
- 시간 차단: transparency(opaque/transparent)
- 메타: etag, created, updated, iCalUID, recurringEventId

### 색상 체계
- 이벤트 색상: colorId 1~11 (Lavender, Sage, Grape, Flamingo, Banana, Tangerine, Peacock, Graphite, Blueberry, Basil, Tomato)
- 캘린더 색상: colorId 1~24 (별도 팔레트, 이벤트와 ID 체계 다름)
- 같은 이름이라도 이벤트/캘린더 간 colorId가 다름 (예: Tomato = 이벤트 11, 캘린더 3)
- Classic/Modern 팔레트 차이 존재 → colors.get() API로 런타임에 조회해야 함
- 이벤트에 colorId 미설정 시 소속 캘린더 색상으로 폴백

### 반복 일정 (RRULE)
- FREQ: DAILY, WEEKLY, MONTHLY, YEARLY
- INTERVAL: 반복 간격 (예: 2주마다 = FREQ=WEEKLY;INTERVAL=2)
- BYDAY: MO,TU,WE,TH,FR,SA,SU (월별은 접두 숫자, 예: 2TU = 둘째 화요일)
- BYMONTHDAY: 월 내 일자
- UNTIL 또는 COUNT: 종료 조건
- EXDATE: 특정 날짜 제외
- 반복 일정 수정 방식:
  - 이번만: 해당 인스턴스만 PATCH → 예외 생성
  - 이후 전체: 원본 RRULE에 UNTIL 추가 + 새 반복 이벤트 생성 (2회 API 호출)
  - 전체: 부모 이벤트 직접 수정
- 반복 일정은 반드시 start.timeZone 필요

### 다중 캘린더
- 사용자당 primary 캘린더 1개 + secondary 캘린더 N개 + 구독 캘린더
- CalendarListEntry: summary, colorId(1~24), backgroundColor/foregroundColor, hidden, selected, accessRole
- 캘린더별 기본 알림(defaultReminders) 설정 가능

### 알림
- useDefault: true → 캘린더 기본 알림 사용
- useDefault: false + overrides[] → 커스텀 알림 (method: email/popup, minutes: 0~40320)
- 최대 5개 알림

### 동기화 메커니즘
- 증분 동기화: syncToken 기반 (최초 전체 조회 → 이후 변경분만 조회)
- syncToken 만료(410 Gone) 시 전체 재동기화
- Push 알림: events.watch()로 웹훅 등록, 변경 발생 시 알림 수신 → syncToken으로 변경분 조회
- 웹훅은 자동 갱신 안 됨 → 만료 전 재등록 필요
- etag: 낙관적 동시성 제어(If-Match/If-None-Match)
- 권장 패턴: 웹훅 + 증분 동기화 조합

### 쿼터
- 일일 1,000,000 쿼리/프로젝트
- 분당 제한 있음 (정확한 수치 비공개, Cloud Console에서 관리)
- 무료

## Google Tasks API

### Task 주요 필드
- title(최대 1024자), notes(최대 8192자)
- due: 날짜만 (시간 무시됨)
- status: needsAction / completed
- completed: 완료 시각 (읽기 전용)
- parent: 상위 Todo ID (읽기 전용, move()로 변경)
- position: 형제 간 정렬 순서 (읽기 전용)
- deleted, hidden: 삭제/숨김 상태

### Tasks API 제한사항 (네이티브 미지원)
- 우선순위(priority) 없음
- 시간 지정 불가 (날짜만)
- 반복 Todo 없음
- 색상/라벨/태그 없음
- 웹훅/Push 알림 없음
- links[] 읽기 전용 (API로 추가 불가)
→ LifeBase는 Google Tasks 기본 필드만 동기화하고, 제품 판단은 기한/고정/완료 흐름으로 단순화한다.

### 동기화 메커니즘
- syncToken 미지원
- 웹훅/Push 알림 미지원
- 폴링 방식: tasks.list()에 updatedMin + showCompleted/showDeleted/showHidden 파라미터
- updated 타임스탬프 기반 변경 감지
- 권장 패턴: 계정별 updatedMin 커서 독립 관리
- 완료 동기화 시 `hidden=true` 항목도 제외하지 않고 반영해야 한다. (Google 완료 항목 누락 방지)

### 쿼터
- 일일 50,000 쿼리/프로젝트 (Calendar의 1/20)
- 폴링 빈도를 보수적으로 설계해야 함
- 무료

### 폴링 주기
- Google Tasks 폴링 주기: 3분
- 5명 × 480회/일 = 2,400 쿼리 (일일 쿼터 50,000의 5%)

## LifeBase 연동 설계 시 주의점

### Calendar vs Tasks 동기화 전략 차이
- Calendar: 웹훅 + syncToken 증분 동기화 (실시간에 가까움)
- Tasks: 폴링 + updatedMin (주기적 동기화, 지연 감수)
- 두 도메인의 동기화 워커를 별도로 설계해야 함

### 색상 매핑
- Google 이벤트 색상(11종)과 캘린더 색상(24종)은 별도 팔레트
- colors.get() API로 hex 값을 런타임 조회, 하드코딩 금지
- LifeBase 자체 색상 체계와 Google 색상 간 매핑 테이블 필요

### Tasks 확장 기능
- 1차 범위는 Google Tasks 호환 필드만 사용한다. (title, notes, due, status, parent)
- 반복/시간/색상은 1차 범위에서 제외한다.
- 고정(Pin)은 로컬 표시/정렬 보조 속성으로만 사용하고 Google Tasks 원본 필드는 확장하지 않는다.

### 다중 계정 운영 규칙
- Source of Truth는 Google Tasks이며, 로컬 DB는 캐시/조회 인덱스 역할로 제한한다.
- 모든 쓰기 작업은 `google_account_id + tasklist_id`를 명시적으로 지정해야 한다.
- 드래그 정렬은 같은 계정 + 같은 Task List 내부에서만 허용한다.
- 다른 Task List 또는 다른 계정으로 이동은 `copy + delete`로 처리한다.
- 계정 재인증 실패는 해당 계정만 읽기 전용 전환하고 다른 계정 동기화는 계속 수행한다.

### 캘린더 계정 식별 규칙 (1차)
- `calendars`에는 `google_account_id`를 저장해 이벤트 색상/필터 판단의 기준으로 사용한다.
- 계정 목록 조회 API(`GET /api/v1/auth/google-accounts`)는 사용자에게 계정 선택 UI 데이터를 제공한다.
- 계정 연결 API(`POST /api/v1/auth/google-accounts/link`)는 현재 로그인 사용자 컨텍스트에서만 동작해야 한다.
- 캘린더 조회 필터는 계정 선택 결과를 캘린더 ID 집합으로 변환해 `calendar_ids`로 적용한다.

### 데이터 요건
- 외부 키(google_account_id, tasklist_google_id, google_id), etag, updated 저장
- Tasks는 계정 단위 `updatedMin` 커서를 저장하고 Calendar는 syncToken을 저장한다.
- 마지막 동기화 시각/상태를 계정 단위로 저장한다.
- 캘린더별 웹훅 채널 ID/만료 시각 관리
- 완료 Todo 정리 정책은 사용자 설정 `todo_done_retention_period`(`1m|3m|6m|1y|3y|unlimited`) 기준으로 계산한다.
