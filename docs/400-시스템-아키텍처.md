# 시스템 아키텍처

## 목적
- 웹/데스크톱(macOS, Windows)/모바일(iOS, Android)을 동시에 지원하는 공통 아키텍처 기준을 정의한다.

## 입력
- 단일 인증: Google OAuth
- 단일 백엔드: 계정/권한/클라우드/동기화 규칙 공통 처리
- 다중 클라이언트: Web, Desktop, Mobile

## 결정
- 아키텍처 스타일: 헥사고날 아키텍처(Ports & Adapters) + 모듈형 모놀리식 + 비동기 워커
- 서버 중심 도메인 규칙: 권한, 용량, 공유, 동기화 충돌 정책은 서버가 최종 판단
- API: REST + WebSocket(동기화 상태/작업 진행 이벤트)
- 백엔드 런타임: Go
- 백엔드 프레임워크: chi(경량 라우터) + 표준 라이브러리 중심
- 데이터베이스: PostgreSQL(권한/메타데이터/동기화 상태)
- 캐시/큐: Redis + Asynq(썸네일, 인덱싱, 동기화 작업)
- 파일 저장소: 로컬 파일시스템(루트 경로 고정) + 스토리지 어댑터 구조
- 미디어 처리: FFmpeg(동영상), libvips 계열(이미지 썸네일)
- 검색: PostgreSQL 인덱스 기반(파일명/경로/태그)
- 호스팅: macOS 호스트 1대 + Thunderbolt 외장 12TB 볼륨
- 운영 정책: 초기 단계에서는 백업 미운영(데이터 손실 리스크를 인지하고 진행)
- API 계층 규칙: 1 API Endpoint = 1 Controller = 1 UseCase
- 책임 분리 규칙: Controller는 입력 검증/응답 매핑만 담당, 비즈니스 로직은 UseCase에만 둔다

## 헥사고날 구조(초안)
- `internal/<module>/domain`: 엔티티, 값 객체, 도메인 규칙
- `internal/<module>/usecase`: 유스케이스(애플리케이션 서비스)
- `internal/<module>/port/in`: 입력 포트(인터페이스)
- `internal/<module>/port/out`: 출력 포트(저장소/외부 API 인터페이스)
- `internal/<module>/adapter/in/http`: 컨트롤러, DTO, 라우트
- `internal/<module>/adapter/out/*`: DB, 파일시스템, Google API 어댑터
- `internal/shared`: 공통 에러, 트랜잭션 경계, 로깅, 관측성

## 썸네일 파이프라인
- 업로드 시 Asynq 워커가 비동기로 썸네일 생성 (이미지: libvips, 비디오: FFmpeg)
- 썸네일 저장: 원본과 같은 파일시스템 내 `.thumbnails/{fileId}/{size}.webp` 경로
- 크기 단계: small(150×150), medium(400×400)
- 포맷: WebP (용량 대비 품질 최적)
- 서빙: 정적 파일 서빙 + `Cache-Control: public, max-age=86400` + `ETag`
- 원본 변경/삭제 시 해당 썸네일도 워커가 함께 삭제/재생성

## 열린 질문
- 대규모 사용자 전환 시 워커 분리 배포 시점
- 스토리지 어댑터(S3 호환) 확장 시점
