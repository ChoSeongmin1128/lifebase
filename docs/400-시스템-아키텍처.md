# 시스템 아키텍처

## 목적
- 웹/데스크톱(macOS, Windows)/모바일(iOS, Android)을 동시에 지원하는 공통 아키텍처 기준을 정의한다.

## 입력
- 단일 인증: Google OAuth
- 단일 백엔드: 계정/권한/클라우드/동기화 규칙 공통 처리
- 다중 클라이언트: Web, Desktop, Mobile

## 결정
- 아키텍처 스타일: 헥사고날 아키텍처(Ports & Adapters) + 모듈형 모놀리식 + 비동기 워커
- 서버 중심 도메인 규칙: 권한, 용량, 공유, 동기화 충돌 정책은 서버가 최종 판단
- API: REST(`/api/v1/` prefix) + WebSocket(동기화 상태/작업 진행 이벤트)
- 페이지네이션: Cursor 기반 (`?cursor=abc&limit=50`)
- 에러 포맷: `{ "error": { "code": "FILE_NOT_FOUND", "message": "..." } }`
- CORS: `api.lifebase.cc`에서 `web.lifebase.cc` origin 허용
- Rate limiting: 클라이언트당 100 req/min
- Health check: `GET /api/v1/health`
- WebSocket 재연결: 지수 백오프 (1s/2s/4s/8s... 최대 30s)
- 백엔드 런타임: Go
- 백엔드 프레임워크: chi(경량 라우터) + 표준 라이브러리 중심
- 데이터베이스: PostgreSQL(권한/메타데이터/동기화 상태)
- 캐시/큐: Redis + Asynq(썸네일, 인덱싱, 동기화 작업)
- 파일 저장소: 로컬 파일시스템(루트 경로 고정) + 스토리지 어댑터 구조
- 미디어 처리: FFmpeg(동영상), libvips 계열(이미지 썸네일)
- 검색: PostgreSQL pg_trgm 기반(파일명/폴더명 부분 일치, 한글 지원)
- 호스팅: Mac Mini(macOS, bare metal) + Thunderbolt 외장 12TB 볼륨
- 파일시스템: APFS (외장 HDD 포함)
- 파일 스토리지 경로: UUID 플랫 구조
  - 원본 파일: `/Volumes/WDRedPlus/LifeBase/data/{user_id}/{uuid_prefix}/{uuid}` (HDD)
  - 썸네일: `/Users/seongmin/lifebase-cache/thumbs/{user_id}/{file_uuid}_{size}.webp` (내장 SSD)
  - 썸네일을 SSD에 두는 이유: HDD 랜덤 I/O 지연(5-15ms/건) → Gallery 50개 동시 로드 시 체감 지연
  - SSD 사용량: 5만 개 기준 small+medium ~10GB (1TB SSD에서 미미)
  - 물리 파일명에 원본 파일명을 사용하지 않음 (한글 NFD/NFC 이슈 회피)
  - 원본 파일명은 DB `files.name`에만 저장
  - UUID 앞 2자리로 디렉토리 분산 (디렉토리당 파일 수 제한)
- DB 마이그레이션: goose (Go 코드 내 실행 가능, NO TRANSACTION 옵션 지원)
- 배포 방식: bare metal (Docker 미사용, macOS에서 Docker의 VM 레이어 I/O 오버헤드 회피)
  - Go 바이너리: launchd로 자동 시작
  - PostgreSQL/Redis: Homebrew로 설치 및 관리
  - HTTPS: Caddy (Let's Encrypt 자동 갱신)
- 도메인: `lifebase.cc` (Cloudflare Registrar 구매, DNS 전용, 프록시 OFF)
  - Cloudflare Free/Pro 플랜은 한국 트래픽을 일본으로 우회 → DNS 전용 모드로 사용
  - 서브도메인 분리: `api.lifebase.cc`, `web.lifebase.cc`
  - DDNS 갱신: 크론 스크립트로 Cloudflare API 호출
- 운영 정책: 초기 단계에서는 백업 미운영(데이터 손실 리스크를 인지하고 진행)
- API 계층 규칙: 1 API Endpoint = 1 Controller = 1 UseCase
- 책임 분리 규칙: Controller는 입력 검증/응답 매핑만 담당, 비즈니스 로직은 UseCase에만 둔다

## 헥사고날 구조(초안)
- `internal/<module>/domain`: 엔티티, 값 객체, 도메인 규칙
- `internal/<module>/usecase`: 유스케이스(애플리케이션 서비스)
- `internal/<module>/port/in`: 입력 포트(인터페이스)
- `internal/<module>/port/out`: 출력 포트(저장소/외부 API 인터페이스)
- `internal/<module>/adapter/in/http`: 컨트롤러, DTO, 라우트
- `internal/<module>/adapter/out/*`: DB, 파일시스템, Google API 어댑터
- `internal/shared`: 공통 에러, 트랜잭션 경계, 로깅, 관측성

## 파일 업로드
- 프로토콜: tus (resumable upload), Go용 tusd 라이브러리 사용
- 단일 파일 크기: 무제한 (할당량 1TB만 제한)
- 동시 업로드: 클라이언트당 5개, 나머지는 클라이언트 업로드 큐(FIFO)에서 대기
- 청크 크기: 기본 5MB
- MIME 타입: 전체 허용, 서버에서 magic bytes로 실제 타입 감지 후 DB 저장

## 썸네일 파이프라인
- 업로드 시 Asynq 워커가 비동기로 썸네일 생성 (이미지: libvips, 비디오: FFmpeg)
- 썸네일 저장: 내장 SSD `/Users/seongmin/lifebase-cache/thumbs/` 경로
- 크기 단계: small(150×150), medium(400×400)
- 포맷: WebP (용량 대비 품질 최적)
- 지원 포맷: JPEG, PNG, WebP, GIF, HEIC/HEIF, 주요 RAW(CR3/ARW/NEF/DNG)
- HEIC: libvips + libheif 의존성 필요
- 서빙: 정적 파일 서빙 + `Cache-Control: public, max-age=86400` + `ETag`
- 원본 변경/삭제 시 해당 썸네일도 워커가 함께 삭제/재생성
- 실패 처리: 3회 재시도(1분/5분/30분 지수 백오프), 최종 실패 시 확장자 placeholder
- 타임아웃: 이미지 30초, 비디오 2분
- EXIF 메타데이터: 업로드 시 파싱 워커가 촬영 날짜/GPS/카메라 정보 추출 → `files.taken_at` 저장

## 비디오 재생
- 기본: Range Request(HTTP 206) 원본 서빙 (탐색 가능)
- 비호환 코덱(HEVC 등): HLS 트랜스코딩 (Asynq 워커, FFmpeg H.264 변환)
- HLS 변환 완료 전: 다운로드 재생 폴백

## 휴지통
- 30일 경과 항목 자동 삭제
- 삭제 워커: 매일 새벽 3시 실행
- 삭제 순서: DB soft delete 확인 → 물리 파일 삭제 → 썸네일 삭제 → DB hard delete
- 부분 실패 시 다음 실행에서 재시도

## 폴더 동기화 (Desktop)
- 파일 감시: macOS FSEvents, Windows ReadDirectoryChangesW (Tauri v2 Rust 사이드카)
- 충돌 판정: 파일 해시(SHA-256) + 타임스탬프 조합. 양쪽 동시 수정 시 충돌
- 충돌 시 정책: 사용자 선택 (로컬 우선/서버 우선/양쪽 보존)
- 무시 패턴: 기본 목록(.DS_Store, Thumbs.db, desktop.ini, ~$*, *.tmp, .git/, node_modules/) + `.lifebaseignore` 사용자 커스텀 (.gitignore 문법)
- 심볼릭 링크: 무시 (실제 파일만 동기화)
- 동기화 큐: 작은 파일 우선, 대용량 파일은 백그라운드
- 동기화 상태: idle → syncing → done / error, 네트워크 끊김 시 paused → 재연결 후 자동 재개

## 모바일 자동 업로드
- 네트워크: WiFi 기본, 셀룰러는 Settings에서 토글
- 중복 감지: 파일 해시(SHA-256)
- 업로드 큐: 최신 사진 우선 (LIFO)
- OS 대응: iOS BGTaskScheduler / Android WorkManager
- 업로드 진행률: 인앱 + 시스템 알림 바

## 공유
- 초대 토큰: 랜덤 32바이트 URL-safe Base64 (10분 만료)
- 용량 차감: 폴더 소유자 할당량에서 차감
- 뷰어 권한: 다운로드 가능
- 공유 해제 시: 상대방 로컬 동기화 데이터 유지 (서버 접근만 차단)
- 무차별 대입 방지: 초대 엔드포인트 rate limiting (IP당 10 req/min)

## 열린 질문
- 대규모 사용자 전환 시 워커 분리 배포 시점
- 스토리지 어댑터(S3 호환) 확장 시점
- 초기 사용자 규모 확대 시 Docker/Linux 전환 시점
